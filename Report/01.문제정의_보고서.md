## STEP 1 — Observation (관찰)

### 1.1 현재 우리가 해결하려는 상황은 무엇인가?

**관찰:**  
한 SW 아키텍트가 **정해진 규칙을 만족하는 4×4 숫자 격자**를 얻고 싶어 하는데, 그 규칙이 무엇인지·어떻게 검증할지·어떤 형태로 결과를 보여줄지는 아직 명시되지 않았다.

| 관찰 항목 | 상태 |
|-----------|------|
| 입력·출력 | 사용자 입력 vs 자동 생성, 표시 매체(콘솔/화면/파일) 불명 |
| “맞는 격자”의 기준 | 행·열·대각 합 동일은 일반적 지식이나, 1~16 각 1회·합 34 등은 미확정 |
| 목적 범위 | 한 번의 정답 vs 생성·검증·탐색·설명 전체 불명 |

**상황 요약:**  
4×4 격자에 숫자를 배치했을 때 여러 방향의 합(또는 다른 불변 조건)이 일치하는지를 다루는 소프트웨어를 만들려는 **의도**가 있으며, 프로젝트 폴더는 교육·실습 맥락의 **빈 시작 상태**이다.

### 1.2 왜 4×4 마방진 문제를 다루는가?

| 관찰 가능한 이유 | 설명 |
|------------------|------|
| 규모·규칙의 명확성 | 3×3보다 조건·해가 늘어 구조 연습에 적합, 5×5보다 검증·시각화 부담 적음 |
| 아키텍트 연습 | 도메인 규칙, 생성 전략, 검증기, 표현(모델/뷰) 분리 연습 |
| 교육 맥락 | 문제 정의 → 설계 → 구현 흐름의 매개 문제 |
| 4×4 특수성 | 짝수 차원 마방진은 홀수 차원과 다른 성질·구성 관점이 있음 |

**미확인:** 비즈니스 요구(게임, 퍼즐 앱) vs 개인 학습 vs 과제 제출 등 최종 동기.

### 1.3 학습·시스템 설계 맥락

| 맥락 | 내용 |
|------|------|
| **학습** | `커서ai교육/dev` 경로, 빈 저장소 → 요구 자체 정의·구현 실습 |
| **시스템 설계** | 제약 만족의 축소판, 검증 vs 생성 분리, 도메인·애플리케이션 경계, n×n 확장 가능성 |
| **관찰되지 않음** | 배포, 다중 사용자, 성능, 영속화, 외부 연동 → 초기 단일 프로세스·로컬 가정 |

### 1.4 표현 전환 (관찰 관점)

| 일상 표현 | 관찰 식 표현 |
|-----------|--------------|
| 4×4 마방진 프로그램을 만든다 | 4×4 격자에 대한 합·배치 규칙을 정의하고, 규칙을 만족하는 배치를 얻거나 검증하는 소프트웨어를 도입하려는 상황 |
| 마방진을 생성한다 | 정의되지 않은 입출력·성공 기준 하에서 “규칙에 맞는 격자”에 대한 기대가 존재함 |
| 문제를 푼다 | 교육·아키텍트 실습에서 제약 문제를 통해 요구 정의와 구조 분리를 연습하려는 의도 |

## STEP 2 — Why #1

### 2.1 질문

**Q:** 왜 마방진을 완성해야 하는가?

### 2.2 가정한 답 (A)

> 4×4 마방진을 완성해야 하는 이유는, 행·열·대각선의 합이 모두 같다는 규칙을 만족하는 배치가 실제로 존재하는지 확인하고, 그 배치를 한눈에 보여 주어 **「규칙이 성립한다」는 것을 검증·설명**할 수 있기 때문이다.

### 2.3 드러나는 불편함

| 불편 | 설명 |
|------|------|
| 목적이 “완성”에 고정 | 과정·다른 해·설명 요구가 완성 이후에야 등장 |
| 존재 확인 vs 표현 혼재 | 검증·탐색과 UI·표시의 우선순위 충돌 |
| “규칙 성립” 범위 모호 | 합만 vs 1~16 각 1회 등 기준 불일치 시 오인 |
| 설명 가능성의 사후화 | 격자만 출력하면 Why #1의 핵심(설명) 미충족 |

### 2.4 구조적 문제

| 문제 | 설명 |
|------|------|
| 단일 목표 vs 다층 가치 | 가치(설명·검증)와 산출물(격자) 분리 실패 시 “출력기”화 |
| 완성 = 종료 | 부분·실패·재실행 상태가 1급 시민이 되지 않음 |
| 검증 위치 | 이유는 검증·설명인데 정의는 생성 중심으로 기울기 쉬움 |
| 4×4 vs 일반화 | “규칙 성립” 학습과 “한 격자 완성” 목표의 충돌 |
| 이해관계자 부재 | 설명 대상·엄격도 기준 부재 |

### 2.5 Why #1 재정리

가정 A는 **인지 층**(일관된 배치가 존재함)과 **신뢰 층**(규칙 만족을 확인 가능함)을 동시에 말한다. “완성”이라는 단어가 이 둘을 뭉개 **생성 모듈만 비대화**하는 편향을 유발할 수 있다.

## STEP 3 — Why #2

### 3.1 질문

**Q:** 왜 단순 계산이 아니라 프로그램으로 구현하는가?

### 3.2 가정한 답 (A)

> 손계산·종이로도 한 번은 맞출 수 있으나, 같은 규칙을 여러 번·여러 조건에서 재현하고, 맞는지 자동 판별하며, 사람의 실수를 줄이고, **「규칙 → 상태 → 판정」** 으로 사고하는 연습을 하려면 실행 가능한 명세(프로그램)가 필요하다.

### 3.3 네 관점 분석

#### 반복 가능성

- **이점:** 동일 절차·검증 순서 재현, 실험·비교 가능  
- **불편:** 사람 반복 시 필기 실수·검산 누락  
- **구조 주의:** 결정적 실행 vs 탐색·다해 — “반복”의 의미 정의 필요

#### 검증 자동화

- **이점:** 행·열·대각 일괄·일관 판정, 불리언 결과  
- **불편:** 검산 피로, 대각 누락, “맞다”의 기준 불일치  
- **구조 주의:** 실패 원인·규칙 버전·도메인/구현 정의 분리

#### 오류 방지

- **이점:** 산술·중복·칸 밀림 등 인지 실수 감소, 파이프라인 통합  
- **불편:** 프로그램 과신으로 검증 생략  
- **구조 주의:** 생성기·검증기 이중 정확성, 사용자 입력 범위

#### 규칙 기반 사고 훈련

- **이점:** 불변식 명시, 도메인·판정·전이 분리 연습  
- **불편:** 종이는 결과 암기에 그칠 수 있음  
- **구조 주의:** 정답 테이블 vs 규칙 엔진 — Why #2와 구현 단순화 충돌

### 3.4 Why #2 종합

프로그램은 계산기가 아니라 **규칙의 반복 실행·자동 판정·실수 감소·명시적 사고**의 매개체이다. 네 가치를 동시에 추구하면 산출물은 **격자 + 규칙 정의 + 검증 결과(및 필요 시 설명)** 를 포함하는 작은 시스템으로 재정의된다.

## STEP 4 — Why #3 (핵심 발견)

### 4.1 질문

**Q:** 왜 이 문제를 TDD 방식으로 설계하려 하는가?

### 4.2 가정한 답 (A)

> 4×4 마방진은 여러 불변 조건이 동시에 성립해야 하는 규칙 문제이므로, **검증 가능한 단위(입력·출력·판정)** 를 먼저 고정한 뒤 생성·표시를 쌓지 않으면 요구가 “완성된 격자”로만 수렴하고 버그·범위 creep를 막기 어렵다.

### 4.3 무엇이 통제되어야 하는가

| 통제 대상 | 내용 |
|-----------|------|
| 규칙의 정의 | 합만 vs 1~16·중복 없음까지 |
| 판정 결과 | 만족/불만족, 위반 종류·위치 |
| 입력 형태 | 빈·부분·완전·잘못된 격자 |
| 출력 의미 | 격자 vs 검증 리포트 vs 성공/실패 |
| 실행 결정성 | 같은 입력 → 같은 판정 |
| 범위 | 4×4 고정 vs n×n |
| 생성 ⊂ 검증 | 성공 주장은 판정기 통과와 동치 |

**핵심:** 통제 대상은 16칸이 아니라 **규칙 엔진의 계약**이다.

### 4.4 불변 조건 계층 (TDD 관점)

| 계층 | 내용 |
|------|------|
| 구조 | 4×4, 완성 시 빈 칸 없음 |
| 집합 | 1~16 각 1회, 범위 |
| 합 | 행·열·주대각·부대각 동일 → 34 (표준 규칙 시) |
| 부분 상태 | 완성 vs 위반 vs 미완 구분 |
| 메타 | 검증 결정성; 생성 성공 ⟹ 검증 통과 |

**주의:** 정답 격자 하나와의 비교만 있으면 TDD가 스냅샷 암기로 퇴화한다.

### 4.5 입력/출력이 명확해야 하는 이유

- 실패가 디버깅·학습 정보가 되려면 계약 필요  
- 도메인 표현과 CLI/GUI/파일 표현 분리  
- 부분 채움·오류 모델(예외 vs Result) 사전 고정  
- 테스트 = 실행 가능한 요구 문장

### 4.6 Why #1~#3 연결

| 단계 | 핵심 |
|------|------|
| Why #1 | 완성 + 규칙 성립·설명 |
| Why #2 | 프로그램 = 반복·검증·오류 방지·규칙 명시 |
| Why #3 | TDD = 통제·불변식·계약을 먼저 고정 |

## STEP 5 — 진짜 문제 정의

### 5.1 표면 문제 정의 (잘못된 정의)

> **「4×4 칸에 숫자를 넣어서 행·열·대각선 합이 같은 마방진을 완성하는 프로그램을 만든다.」**

| 한계 | 설명 |
|------|------|
| 산출물 중심 | 규칙 충족·설명·재검증 후순위 |
| 규칙 모호 | 숫자 집합·대각·완성 기준 없음 |
| 역할 혼재 | 생성·검증·표시·설명 일체 |
| 검증 생략 가능 | 맞는 것처럼 보이는 잘못된 성공 |
| TDD·아키텍트 맥락 상실 | 통제·불변식·계약 의도 부재 |

### 5.2 개선된 문제 정의 (정확한 정의)

> **「4×4 격자에 대한 마방진 규칙을 명시적으로 정의하고, 임의의 배치(완전·부분)가 그 규칙을 만족하는지 일관되게 판정할 수 있으며, 규칙을 만족하는 완전한 배치를 얻었을 때 그 결과가 판정기와 동일한 기준으로 재확인 가능한 형태로 제공되는 시스템을 정의한다.」**

#### 정의 요소

| 요소 | 내용 |
|------|------|
| 규칙 범위 | 4×4; 1~16 각 1회; 행·열·주대각·부대각 합 동일 |
| 판정 | 동일 기준으로 반복 판정; 필요 시 위반 내용 |
| 완전 배치 | 16칸 채움 + 모든 규칙 만족 |
| 부분 배치 | (요구 시) 조기 위반 판별 |
| 신뢰 관계 | 완전 배치 제공 ⟹ 판정기 통과 |
| 경계 | 채움·표시 방식은 별도 계약, **판정 기준은 하나** |

#### 성공 기준 (개념)

- 규칙을 설명 가능  
- 맞는/틀린 배치 구분 가능  
- 얻은 완전 배치를 판정기로 재확인 시 일치  
- (선택) 반복 실행 시 판정 의미 불변  

#### 의도적 비포함

특정 UI·언어·성능; 유일 정답/전체 해; n×n 일반화(현재 4×4).

### 5.3 핵심 Invariant

#### 구조

| ID | 불변 조건 |
|----|-----------|
| I-1 | 정확히 4행 4열 |
| I-2 | 완전 배치 시 모든 칸에 값 존재 |

#### 값·집합

| ID | 불변 조건 |
|----|-----------|
| I-3 | 각 값 ∈ [1, 16] |
| I-4 | 1~16 각 정확히 1회 |

#### 합

| ID | 불변 조건 |
|----|-----------|
| I-5 | 네 행의 합 동일 |
| I-6 | 네 열의 합 동일 |
| I-7 | 주대각·부대각 합이 행·열과 동일 |
| I-8 | I-3·I-4 성립 시 공통 합 = 34 |

#### 시스템·신뢰

| ID | 불변 조건 |
|----|-----------|
| I-9 | 동일 배치·규칙 → 동일 판정 (결정적) |
| I-10 | 완전 배치 제공 주장 ⟹ I-1~I-8 판정기 확인 |

#### 부분 배치 (선택)

| ID | 불변 조건 |
|----|-----------|
| I-11 | 부분 배치: 미완성 vs 이미 위반 구분 |

### 5.4 훈련하려는 사고 능력

| 능력 | 설명 |
|------|------|
| 규칙 명시화 | 직관을 검사 가능한 조건 집합으로 |
| 불변식 분해 | 구조/집합/합/메타 계층 |
| 계약·경계 | 판정·획득·표현 책임 분리 |
| 증거 기반 신뢰 | 판정 가능한 근거로 성공 정의 (I-10) |
| 반복 가능한 사고 | 일회성 계산·암기 의존 탈피 |
| 테스트 주도적 문제 정의 | 기능 목록보다 만족/위반 예시·출력 의미 우선 |
| 범위 통제 | 4×4 집중, 표면 정의에 끌리지 않기 |
