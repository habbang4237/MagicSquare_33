## 1. 세션 요약

Dual-Track UI + Logic TDD 코치 역할로, SSOT(PRD·Report/02·`.mdc`)를 참조하여 **RED 테스트 설계표만** 작성했다.

| 순서 | 사용자 요청 | 결과 |
|------|-------------|------|
| 1 | TRACK A/B RED 설계표 작성 (코드·파일·pytest 금지) | UI·Logic RED 표 + 검수 체크리스트 출력 |
| 2 | Report 생성 + Prompt Transcript Export | 본 보고서, `Prompt/09.*` |

**세션 금지 사항 (준수):**

- 구현 코드, 테스트 코드, 스켈레톤, 파일 저장(1차 턴)
- GREEN / REFACTOR
- 클래스·파일 구조 확정

**고정 계약 (설계 기준):**

| 항목 | 내용 |
|------|------|
| 입력 | `int[4][4]`, `0`=빈칸(정확히 2개), 값 ∈ `{0}∪[1,16]`, 비零 중복 금지 |
| 출력 | `int[6]` = `[r1,c1,n1,r2,c2,n2]`, 좌표 **1-index** |
| 마방 상수 | `M = 34` (`MagicConstant` SSOT) |
| Boundary 실패 | `ErrorResponse { code, message }` |
| 입력 실패 | `SolvePuzzleUseCase.execute` **0회** (AC-FR01-5) |
| ECB | `boundary → control → entity`; Boundary는 Solver/Validator **직접 호출 금지** |
| Dual-Track | Track A: Domain Mock 허용 / Track B: **Domain Mock 금지** |

## 3. 설계 범위

프롬프트 TRACK A·TRACK B 명세에 따라 다음을 RED 설계에 포함했다.

| 구분 | 포함 Test-ID | 비고 |
|------|--------------|------|
| **입력 계약** | U-T00, U-T01a~f, U-T02a~b, U-T03a~b, U-T04 | PRD §13 `E_*` 코드·메시지 |
| **흐름 격리** | AC-FR01-5 | invalid 입력마다 `execute` 0회 |
| **출력 계약** | U-T05, U-T07 | 입력 통과 + Mock UseCase |
| **Domain** | D-T03, D-T05, D-T04, D-T04-R/C/D, D-T01, D-T02, D-T13, D-T10, D-T11 | Mock 금지 |

**본 설계표에서 의도적으로 제외(후속 슬라이스):**

- U-T06 (`E_NO_SOLUTION` — Domain Mock `NO_VALID_COMPLETION`)
- U-T08 (전체 `E_*` 메시지 회귀)
- D-T12, D-T14, D-T15, D-T16
- 통합 IT-*

**슬라이스 축소 시:** UI 표에서 `U-T00`(+ 선택 `U-T01f`)만 남기면 AC-FR-01-01(크기) 슬라이스와 동일.

## 4. Track A — Boundary RED 요약

### 4.1 입력 검증 (Domain 미호출)

| Test-ID | 시나리오 | 기대 code | AC-FR |
|---------|----------|-----------|-------|
| U-T00 | `grid=None` | `E_SIZE` | AC-FR01-1, AC-FR01-5 |
| U-T01a~f | 4×4 아님 (`[]`, 열0, 3×4, 4×3, 5×5, `invalid_size`) | `E_SIZE` | AC-FR01-1, AC-FR01-5 |
| U-T02a | 빈칸 0개 | `E_EMPTY_COUNT` | AC-FR01-2, AC-FR01-5 |
| U-T02b | 빈칸 3개 | `E_EMPTY_COUNT` | AC-FR01-2, AC-FR01-5 |
| U-T03a | 값 `17` | `E_RANGE` | AC-FR01-3, AC-FR01-5 |
| U-T03b | 값 `-1` | `E_RANGE` | AC-FR01-3, AC-FR01-5 |
| U-T04 | 비零 `5` 중복 | `E_DUPLICATE` | AC-FR01-4, AC-FR01-5 |

**검증 관심 순서:** 크기 → 빈칸 수 → 값 범위 → 중복. 다중 위반(예: 3×4 + `17`)은 **첫 위반 `E_SIZE`** (DN-03).

**When (호출 대상):** `PuzzleBoundary.receive(grid)` / `InputContractValidator.validate(grid)`

**Given 앵커 (PRD §16.4):**

| Label | Grid 요약 |
|-------|-----------|
| `invalid_size` | `[[1,2,3],[4,5,6],[7,8,9]]` |
| `invalid_blank_1` | 4×4, `count(0)==1` |
| `invalid_range` | 셀 `17` 또는 `-1` |
| `duplicate_five` | `5` 두 칸 |

### 4.2 출력 계약 (유효 입력 + Mock)

| Test-ID | 시나리오 | 기대 |
|---------|----------|------|
| U-T05 | Mock `execute` → 6원 배열 | `len(result)==6`, `execute` 1회 |
| U-T07 | Mock `r1=0` | `E_OUTPUT_FORMAT` |

### 4.3 흐름 격리

| Test-ID | 시나리오 | 기대 |
|---------|----------|------|
| AC-FR01-5 | U-T00~U-T04 전체 invalid Given | `execute.call_count==0` |

**Expected RED Failure (공통):** `ImportError` / `ModuleNotFoundError`(미구현), `AssertionError`(wrong code·message·호출 횟수), generic `Exception` 반환.

| D-T02 | `PuzzleSolver` | `reverse_only_success` | `[r1,c1,m2,r2,c2,m1]` | I-D4~I-D7 |

**Expected RED Failure (공통):** `ImportError`, `AssertionError`(좌표·벡터·`is_valid` 불일치), 잘못된 `DomainError` 코드.

## 6. RED 설계 검수

| 항목 | 결과 |
|------|------|
| Boundary 실패는 `ErrorResponse`(generic Exception 아님) | ✓ |
| invalid → `execute` 0회 포함 | ✓ |
| 입력 검증 vs 출력 계약 분리 | ✓ |
| Logic Track에 Domain Mock 없음 | ✓ |
| Test-ID·AC-FR·I-D* 추적 가능 | ✓ |
| 코드/스켈레톤/파일 생성 없음 (1차 턴) | ✓ |
