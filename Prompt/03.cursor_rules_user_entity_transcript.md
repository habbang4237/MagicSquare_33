## 1. `.cursorrules` vs `.cursor/rules/*.mdc` — 무엇을 쓸까?

| 방식 | 위치 | 특징 | MagicSquare에 적합한가 |
|------|------|------|------------------------|
| **`.cursorrules`** (레거시) | 프로젝트 루트 단일 파일 | 항상(또는 거의 항상) 전체 적용, glob·역할 분리 없음, 길어지면 컨텍스트 낭비 | ❌ 단독 사용 비추천 |
| **`.cursor/rules/*.mdc`** (권장) | `.cursor/rules/` 아래 여러 파일 | YAML frontmatter(`alwaysApply`, `globs`, `description`), **파일·레이어별** 규칙, Rule Picker에서 선택 가능 | ✅ **이 방식 선택** |

**권장 결론:** MagicSquare는 **`.cursor/rules/*.mdc`만** 쓰고, 루트 `.cursorrules`는 **두지 않거나** 5~10줄짜리 “인덱스”만 둡니다(예: “모든 작업은 `.cursor/rules/`를 따른다. 상세는 `Report/` 참조”).

이유는 조건이 서로 다른 성격이기 때문입니다.

- **항상 지켜야 할 것:** RED → GREEN, 테스트 약화 금지, `print()` 금지  
- **파일 종류별:** `tests/` vs `src/domain/` vs `src/boundary/`  
- **Dual-Track:** UI 테스트는 Domain Mock, Domain 테스트는 Mock 없음  

한 파일에 몰면 200줄 넘기 쉽고, Cursor Rule 가이드(대략 **규칙당 ~50줄, 한 관심사당 한 파일**)와도 맞지 않습니다.

## 2. 규칙 파일 하나의 표준 섹션(템플릿)

각 `.mdc`는 **frontmatter + 본문** 구조를 씁니다.

```yaml
---
description: 한 줄 요약 (Rule Picker용)
globs: src/domain/**/*.py   # 파일별 규칙일 때
alwaysApply: false            # 전역 워크플로일 때 true
---
```

본문은 아래 **고정 섹션 순서**를 추천합니다(에이전트가 스캔하기 쉽습니다).

| 섹션 | 내용 |
|------|------|
| **1. Scope** | 이 규칙이 적용되는 경로·레이어·Dual-Track 중 어느 트랙인지 |
| **2. Non‑negotiables** | MUST / MUST NOT (짧은 불릿, 5~10개) |
| **3. Workflow** | Dual-Track TDD, RED 없이 구현 금지 등 **순서** |
| **4. Architecture** | ECB에서 이 폴더의 책임·의존 방향 |
| **5. Code & types** | Python 3.14, PEP 8, type hints, 네이밍 |
| **6. Testing** | pytest, AAA, Mock 정책, 커버리지/계약 ID(D-T01, U-T01 등) |
| **7. Examples** | ❌ BAD / ✅ GOOD 각 1~2개 (짧게) |
| **8. References** | `Report/02...md` §절, Invariant ID 링크 |

**반드시 “행동 가능한 문장”으로 쓰기:**  
“type hints 권장” ❌ → “public 함수·메서드 시그니처에 반환 타입 포함, `Any` 금지(예외: JSON 파싱 1곳)” ✅

## 3. MagicSquare용 규칙 파일 나누기(권장 6~7개)

설계 문서의 **Logic Track / UI Track / Data / Integration**과 ECB를 그대로 반영합니다.

```
.cursor/rules/
├── 00-magicsquare-core.mdc          # alwaysApply: true
├── 01-tdd-dual-track.mdc            # alwaysApply: true (또는 tests+src globs)
├── 10-python-standards.mdc          # globs: **/*.py
├── 20-architecture-ecb.mdc          # globs: src/**/*.py
├── 30-domain-entity.mdc             # globs: src/**/entity/**, domain value objects
├── 31-domain-services.mdc           # globs: src/**/domain/** (services)
├── 40-control-usecases.mdc          # globs: src/**/application/**, control/**
├── 50-boundary-ui.mdc               # globs: src/**/boundary/**
├── 60-data-repository.mdc           # globs: src/**/data/**, infrastructure/**
└── 70-pytest-aaa.mdc                # globs: tests/**/*.py
```

구현 전이라 `src/` 경로가 아직 없어도, **지금 문서의 패키지명을 미리 고정**해 두면 나중에 규칙을 안 옮겨도 됩니다. 설계서와 맞추려면 예시는 다음과 같습니다.

| ECB | 설계서 대응 | glob 예시 |
|-----|-------------|-----------|
| **Entity** | `Grid`, Value Objects | `src/magicsquare/domain/**/*.py` (entity/vo) |
| **Control** | `SolvePuzzleUseCase`, Application | `src/magicsquare/application/**/*.py` |
| **Boundary** | Input/Output Contract, `ErrorResponse` | `src/magicsquare/boundary/**/*.py` |
| **Data** (ECB 밖 포트) | `MatrixRepository` | `src/magicsquare/data/**/*.py` |

파일이 너무 잘리면 **30+31 → `30-domain-layer.mdc` 하나**로 합쳐도 됩니다. 핵심은 **Boundary / Domain / Control 의존 규칙**이 한 파일에 섞이지 않게 하는 것입니다.

## 4. 파일별로 넣을 내용(요약 매트릭스)

| 파일 | `alwaysApply` | 담을 Non‑negotiables |
|------|---------------|----------------------|
| **00-core** | `true` | 프로젝트 목적, I-D1~I-D12·고정 계약(2빈칸, `int[6]`, M=34), “판정기 = 신뢰 원천”, `Report/` 우선 |
| **01-tdd-dual-track** | `true` | RED 없이 prod 코드 금지; 실패 테스트·기대값 먼저; 테스트 약화·`skip`·과도한 `xfail` 금지; UI Track = Domain Mock / Domain Track = Mock 없음 |
| **10-python** | `**/*.py` | Python **3.14**, PEP 8, type hints 필수, `print()` 금지(로깅은 boundary 밖·테스트에서 stdout 검사 안 함) |
| **20-architecture-ecb** | `src/**` | 의존 방향: Boundary → Control → Domain; Domain은 IO/UI 모름; 좌표 1-index는 Boundary 대외 계약만 |
| **30-domain** | domain glob | Entity는 검증 로직 최소(`Grid` 설계대로); Domain Service가 I-D* 검증; `DomainError` 코드 고정 |
| **40-control** | application glob | Use case = orchestration만; DomainError → Boundary 코드 매핑은 Boundary 또는 thin mapper |
| **50-boundary** | boundary glob | 입력 스키마 실패 시 Domain 미호출; UX-3 메시지 바이트 일치; 성공 `int[6]`만 |
| **60-data** | data glob | `MatrixRepository` 계약; corrupt/not found 코드; Domain 로직 복제 금지 |
| **70-pytest-aaa** | `tests/**` | Arrange–Act–Assert 명시(주석 또는 빈 줄); 테스트 ID(D-T01, U-T01) 주석; parametrized는 시나리오 이름 명확히 |

**Dual-Track TDD**는 `01-tdd-dual-track.mdc`에 워크플로 다이어그램을 짧게 넣는 것이 좋습니다.

```text
[UI Track]  test_boundary → Mock Domain → 계약·메시지·호출 1회
[Logic Track] test_domain → real objects → I-D*, UC-D4
```

**RED 없이 구현 금지**는 `01`과 `70` 양쪽에 같은 한 줄을 넣어도 됩니다(중복 1문장은 허용, 긴 설명은 한곳만).

## 5. `alwaysApply` vs `globs` 배분 원칙

| 적용 방식 | 쓰는 규칙 |
|-----------|-----------|
| **`alwaysApply: true`** (2~3개만) | 00-core, 01-tdd-dual-track, (선택) 짧은 “에이전트 행동: 구현 요청 시 먼저 테스트 파일 제안” |
| **`globs`로 분기** | Python 스타일, ECB 레이어, pytest AAA, data/boundary |

**주의:** `alwaysApply`를 5개 이상 두면 매 대화마다 토큰이 커집니다. MagicSquare는 **전역 2개 + 나머지 glob**이 균형이 좋습니다.

## 6. 설계 문서(`Report/`, `Prompt/`)와의 관계

Rule 본문에 설계 전체를 복붙하지 말고:

- **00-core** → “불변조건·계약은 `Report/02.DualTrack_CleanArch_TDD_설계.md` §고정 계약, §1.2, §2.2가 정본”
- **레이어별 규칙** → 해당 §번호만 인용 (예: Boundary → §2.2, UX-1~5)

이렇게 하면 Report가 바뀔 때 Rule은 **참조 한 줄**만 고치면 됩니다.

선택: 루트 `AGENTS.md`에 “디렉터리 맵 + 브랜치(`spec`/`develop`) + Dual-Track 순서”만 두고, 상세 강제는 `.cursor/rules/`에 둡니다. Cursor는 둘 다 읽을 수 있지만, **강제력 있는 MUST NOT**은 `.mdc`에 두는 편이 낫습니다.

## 7. 피해야 할 안티패턴

| 안티패턴 | 이유 |
|----------|------|
| 규칙 1개에 300줄+ 설계서 요약 | 컨텍스트 비용↑, 에이전트가 핵심을 놓침 |
| `.cursorrules` + `.mdc`에 동일 내용 중복 | 충돌·드리프트 |
| “PEP8 준수”만 있고 예시 없음 | `print`, `Any`, 테스트 `assert True` 같은 위반이 반복됨 |
| glob 없이 레이어 규칙을 `alwaysApply` | Boundary 작업할 때 Domain 규칙까지 항상 로드 |
| 테스트 약화 금지를 말만 하고 예시 없음 | `pytest.raises` 삭제, 기대값 완화를 막기 어려움 |

**70-pytest-aaa**에 넣을 좋은 예시 한 줄:

- ❌ 실패하는 assertion 제거 후 구현  
- ✅ RED: `assert solver.solve(grid) == [1,2,3,4,5,6]` → GREEN → REFACTOR

```yaml
tdd_rules:
  red_phase:
    must:
      - 신규·변경 요구사항마다 실패하는 테스트를 먼저 작성한다
      - 테스트는 Arrange-Act-Assert(AAA) 구조로 작성한다
      - 기대값·실패 이유가 요구사항·계약(Report/02 설계)과 일치하는지 명시한다
    run: pytest로 해당 테스트를 실행해 FAIL을 확인한다
    proceed_when:
      - 대상 테스트가 의도한 이유로 실패한다(NotImplemented, AssertionError, 기대 DomainError 등)
      - 아직 해당 동작을 만족하는 프로덕션 코드가 없거나 불완전하다
    block_next_phase_if:
      - 테스트가 통과한다(아직 RED가 아님)
      - import 오류·문법 오류·환경 오류로 실패한다(요구 RED가 아님)
      - 실패 테스트 없이 구현·리팩터링을 시작했다
  green_phase:
    must:
      - RED를 확인한 뒤에만 최소한의 프로덕션 코드를 추가·수정한다
      - 현재 실패 테스트를 통과시키는 데 필요한 코드만 작성한다
    run: pytest로 대상 테스트가 PASS인지 확인한다
    forbid:
      - 리팩터링(이름 변경, 추출, 구조 재배치, 중복 제거)
      - RED와 무관한 기능·파일 추가
      - 테스트 기대값·assertion 완화·삭제·skip·xfail로 통과시키기
    proceed_when:
      - 대상 테스트(및 해당 범위 기존 테스트)가 모두 PASS한다
  refactor_phase:
    must:
      - GREEN 확인 후에만 구조·가독성·중복을 개선한다
      - 동작·공개 API·입출력 계약은 변경하지 않는다
    run: pytest 전체(또는 해당 트랙) 실행 후 PASS 유지를 확인한다
    coverage:
      - 리팩터 전후 branch/line 커버리지가 감소하지 않아야 한다
      - 설계서 기준(UI Boundary ≥85% branch 등) 미달 시 리팩터 완료로 보지 않는다
    forbid:
      - 테스트 없이 동작 변경
      - 커버리지 하락을 mask하는 테스트 삭제·약화
```

## 1. YAML 문법

- **구조적 파싱 오류는 보이지 않음** (들여쓰기, 리스트, 주석 `#` 구분선 형태는 일반 YAML로 유효해 보임).
- 다만 이 환경에서 PyYAML 등으로 검증 실행은 하지 못했습니다.
- **Cursor 동작 관점:** `.cursorrules`는 공식 권장 형식(`.cursor/rules/*.mdc`)과 달리, Cursor가 **YAML 스키마로 파싱한다는 보장이 없습니다.** 대부분 **평문 규칙**으로 읽힙니다. `project:`처럼 값이 비어 있는 키는 구조화된 설정이 아니라 **빈 줄에 가까운 노이즈**로만 전달될 수 있습니다.

## 2. 누락된 필수 섹션

최상위 8개 키는 모두 있으나, **내용이 있는 것은 `tdd_rules`뿐**입니다.

| 섹션 | 상태 | 문제 |
|------|------|------|
| `project` | 비어 있음 | 프로젝트 목적·정본 문서(`Report/`) 참조 없음 |
| `code_style` | 비어 있음 | Python 3.14, PEP 8, type hints 등 미정의 |
| `architecture` | 비어 있음 | ECB, Dual-Track, 레이어 의존 방향 없음 |
| `testing` | 비어 있음 | pytest·AAA·커버리지 정책이 `tdd_rules`에만 일부 존재 |
| `forbidden` | 비어 있음 | `print()` 금지, 테스트 약화 금지 등이 섹션에 없음 (`green_phase.forbid`에만 일부) |
| `file_structure` | 비어 있음 | 패키지/레이어 경로 없음 |
| `ai_behavior` | 비어 있음 | 에이전트 작업 순서·금지 행동 없음 |

추가로, 처음 요구하셨던 **Dual-Track(UI Mock / Domain 무 Mock)** 규칙은 `tdd_rules` 어디에도 없습니다.

## 3. `tdd_rules` ↔ `forbidden` 충돌

- **`forbidden`이 비어 있어 문면상 직접 충돌은 없습니다.**
- 다만 **역할 중복·충후 불일치 위험**이 있습니다.
  - 테스트 약화·`skip`·`xfail` 금지 → 현재 `green_phase.forbid`에만 있음.
  - `forbidden`을 나중에 채울 때 같은 항목을 다시 넣으면 **중복**, 한쪽만 완화하면 **불일치**가 날 수 있습니다.
- `refactor_phase.forbid`의 “테스트 삭제·약화”와 `green_phase.forbid`의 “assertion 완화·삭제·skip·xfail”은 서로 **범위가 다릅니다** (리팩터 단계 vs 그린 단계). `forbidden`에 통합 금지를 넣을 때 단계별 예외를 정하지 않으면 해석 충돌 가능성이 있습니다.

**`tdd_rules` 내부 실무 충돌(참고):**  
`red_phase.block_next_phase_if`의 “import·문법·환경 오류로 실패 → 다음 단계 금지”는, 모듈/클래스가 아직 없을 때 테스트를 먼저 쓰는 **일반적인 TDD 첫 RED**(예: `ModuleNotFoundError`)와 맞지 않을 수 있습니다. `forbidden`과의 충돌은 아니지만 규칙 해석상 모순 소지가 있습니다.

## 4. Cursor AI가 실제로 따르기 어려운 규칙

### `ai_behavior`
- **내용이 없어**, 이 섹션 기준으로 검토할 규칙이 없습니다.
- 대신 **`tdd_rules`에 ai_behavior에 넣기 적합한 실행·게이트 규칙**이 들어가 있으며, 아래는 에이전트가 **매 턴 자동 보장하기 어렵습니다.**

| 위치 | 규칙 | 문제 |
|------|------|------|
| `red_phase.run` / `green_phase.run` / `refactor_phase.run` | pytest 실행·FAIL/PASS 확인 | Ask 모드·도구 미호출 시 실행 불가; 사용자가 “코드만 달라”고 하면 생략되기 쉬움 |
| `proceed_when` / `block_next_phase_if` | RED/GREEN 단계 전환 조건 | Cursor에 **단계 상태머신이 없음**; 규칙 위반 여부를 시스템이 차단하지 않음 |
| `refactor_phase.coverage` | 리팩터 전후 커버리지 비교·UI Boundary ≥85% | `coverage` 설정·기준선·레이어별 측정이 없으면 **검증 불가**; 수치는 에이전트가 추정할 수 없음 |
| `refactor_phase.must` | “동작·공개 API·계약 변경 없음” | 리팩터 후 **자동 계약 검증** 없이는 준수 여부를 AI만으로 확정하기 어려움 |

### 빈 섹션으로 인한 동작 공백
- `ai_behavior`가 비어 있어, **“구현 요청 시 테스트부터”, “RED 없이 구현 금지”** 같은 전역 행동 강제가 파일 전체에 없습니다 (`tdd_rules`에만 부분 반영).
- `forbidden`의 `print()` 금지 등도 없어, **금지 목록과 TDD 단계 규칙이 분리·공백** 상태입니다.
